# product.md

A semantic specification for living product documents.

PRODUCT.md describes a product line — its positioning, specifications, production capacity, distribution, and current state — in a single, persistent document written in natural language. It has no end state: it lives as long as the product line exists.

## What it is

Most documents about products are written to be finished — a business plan gets approved, a proposal gets funded, a project plan gets completed. A PRODUCT.md is different. It is a reference document that persists for the life of the product line, continuously updated to reflect reality.

It answers two questions at once: **What is this product?** and **Where does it actually stand right now?**

## Quick overview

A PRODUCT.md has one summary line and five sections, ordered from most stable to least stable:

0. **Summary** — one sentence: why this product line exists and what it looks like now
1. **Positioning** — what it is, who it is for, what it is not
2. **Specifications** — what one unit of the product looks like
3. **Capacity and Distribution** — how it is made and how it reaches audiences
4. **Current Situation** — the gap between intent and reality right now
5. **Related Projects** — links to active project documents (index only, no duplicated content)

## Design principles

- **No end state.** The document persists; it is never "completed."
- **Stability gradient.** Positioning rarely changes; current situation changes constantly.
- **Honesty over aspiration.** Report reality, not wishes.
- **Semantic, not structural.** The spec defines what to say, not how to format it.

## Prior art

The specification draws from television production bibles (living reference documents for ongoing series), game design documents (living blueprints that evolve with development), and Anne Zeiser's transmedia marketing plan framework (stability gradient: goals rarely change, strategies occasionally change, tactics change often).
